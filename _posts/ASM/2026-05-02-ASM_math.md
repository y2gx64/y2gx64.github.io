## Introduction
This post covers the basic arithmetic functions in assembly.

## INC
The `INC` instruction increases the operand by one. The syntax for it is:

```
INC operand
```

It can be used in the following:
```
INC RAX 
INC AX
INC [number] ;increases the variable number by 1
```

### Example 1

```
section .data
  num1 dw 2
section .text
  global  _start


_start: 
  mov ax, 5
  inc ax ;ax to 6, bp1 here
  inc [num1] ;bp2 here, ax shld be 6

exit: mov rax, 60 ;bp3 here, num1 should be 3
      syscall
```

At the second breakpoint, the register `AX` contained `6` due to the increment earlier on.

![](assets/posts/ASM/math/inc2.png)

At the third breakpoint, the variable `num1` contained `3`:

![](assets/posts/ASM/math/inc3.png)


## DEC
The `DEC` instruction is used for decrementing an operand by one. The syntax for it is:

```
DEC operand
```

It can be used in the following:

```
DEC RA
DEC AX
DEC [number] ;decreases the variable number by 1
```

## Example 1

```

section .data
  num1 dw 2
section .text
  global  _start


_start: 
  mov ax, 5
  dec ax ;ax to 4, bp1 here
  dec [num1] ;bp2 here, ax shld be 4

exit: mov rax, 60 ;bp3 here, num1 should be 1
      syscall

```

At the first breakpoint, the register `AX` contained `5`:

![](assets/posts/ASM/math/dec1.png)

At the second breakpoint, the register `AX` contained `4`:


![](assets/posts/ASM/math/dec2.png)

## ADD/SUB
The ADD and SUB instructions are used for performing simple addition/subtraction of binary data. The syntax for it is:

```
ADD/SUB destination, source
```

This can take place in the following combinations:
1.  Register to register
2.  Memory to register
3.  Register to Memory


### Example

```

section .data
  num1 dw 4 ;mem
  num3 dw 1
section .bss
  num2 resb 1
section .text
  global  _start


_start:
  mov rbx, 6
  add rax,rbx ;register to register
  xor rax,rax ;clear rax here ;bp 1: rax = 6
  add rax, num1 ;memory to register. 
  add [num3], rax ;register to memory bp 2: rax = num1

  
exit:  
  mov rax,60 ;bp 3: num3 = 5
  syscall

```

At the first breakpoint, i can see that `RAX` is `6`:
![](assets/posts/ASM/math/add1.png)


At the third breakpoint, i can see that `num3` has the value `5`:

![](assets/posts/ASM/math/add4.png)

## IMUL/MUL
`IMUL`` is for signed integers while `MUL` is for unsigned integers. The syntax for it is:

```
IMUL multiplier 
```

As for where the multiplicand should be stored:

1. If the multiplicand is 1 byte aka 8 bits, it should be in the `AL` register. The product is in AX. High-order 8 bits of the product is stored in AH and the low-order 8 bits are stored in AL.
2. If the multiplicand is 2 bytes aka 16 bits, it should be stored in the `AX` register.The resultant product is a doubleword aka 4 bytes, the high-order (leftmost) portion gets stored in `DX` and the lower-order (rightmost) portion gets stored in `AX`
3. If the multiplicand is 4 bytes aka 32 bits, it should be stored in the `EAX` register.  The high-order portion is stored in `EDX` while the lower-order portion is stored in `EAX`.
4. If the multiplicand is 8 bytes aka 64 bits, it should be in the `RAX` register.The high-order portion is stored in `RDX` while the lower-order portion is stored in `RAX`.

### Example 1 

```

section .data
  multiplier db 3
section .text
  global _start

_start:
  mov al, 2 ;bp2 - RAX should be 2
  mul byte [multiplier]

exit: mov rax, 60 ;bp1 - rax should be 6
      syscall

```

At the first breakpoint i can see that `RAX` has the value `3` :

![](assets/posts/ASM/math/mul1-1.png)

At the second breakpoint, i can see that the register `RAX` now has the value `6` which is the product from the previous operation:

![](assets/posts/ASM/math/mul1-2.png)


### Example 2

```


section .data
  multiplicand dw 2
  multiplier dw 4
section .text
  global _start

_start:
  mov ax, WORD [multiplicand] ; move exactly 2 bytes to AX
  mul WORD [multiplier] ;bp1 - RAX should be 2

exit: mov rax, 60 ;bp2 - RAX should have 8
      syscall
```

At the first breakpoint, i can see that register `RAX` has the value `2` which is the value of the variable `multiplicand`:

At the second breakpoint, i can see that register `RAX` has the value `8` which is the product from the previous operation

## IDIV/DIV 
`IDIV` is for signed integers while `DIV` for unsigned integers. The division operation generates a quotient and a remainder. The syntax for it is:
```
DIV/IDIV divisor

```
As for where the quotient and remainder are stored:

1. If the divisor is 1 byte, the dividend is assumed to be 2 bytes and stored in the `AX` register. The quotient is then stored in the `AL` register with the remainder being in the `AH` register.

2. If the divisor is 2 bytes, the dividend is assumed to be 4 bytes and stored in the `DX` and `AX` registers. The high-order 16 bits are in `DX` and the low-order 16 bits are in `AX`. The quotient is then stored in the `AX` while product in the `DX` register.

3. If the divisor is 4 bytes, the dividend is assumed to be 8 bytes and stored in the `EDX` and `EAX` registers.  The high-order 32 bits are in `EDX` and the low-order 32 bits are in `EAX`. The 32-bit quotient goes to the `EAX` register and the 32-bit remainder goes to the `EDX` register.

### Example 1
```
section .data
  dividend dw 7
  divisor db 2
section .text
  global _start

_start:
  mov ax, WORD [dividend]
  DIV byte [divisor] ;bp1 - RAX is 7

exit: mov rax, 60 ;bp2 - AL is 3 and AH is 1
      syscall
```

At the first breakpoint, i can see that `RAX` has the value `7` as the value of `dividend` was moved to it:
![](assets/posts/ASM/math/div1-1.png)


At the second breakpoint, as the quotient is `3`, the register `AL` has the value `3` while `AH` is 1 due to the remainder being `1`:

![](assets/posts/ASM/math/div1-2.png)

### Example 2

```

section .data
  divisor dw 2 ;2 bytes AKA 16 bits
section .text

  global _start

_start:
  mov ax, 9 ;low end
  xor dx, dx ;high end is 0
  DIV WORD [divisor] ;bp1 - RAX is 9

exit: mov rax, 60 ;bp2 - AX is 4 and DX is 1
      syscall

```


At the first breakpoint, i can see that the register `AX` has the value `9` which is the low-order 16 bits. The register `dx` has the value `0` which is the high-order bits. This creates a dividend of `9`:

![](assets/posts/ASM/math/div2-1.png)

At the second breakpoint, the register `AX` has the value `4` while `DX` has the value `1`:

![](assets/posts/ASM/math/div2-2.png)

### Example 3

```
section .data
  divisor dq 2 ;4 bytes aka 16 bits
section .text

  global _start

_start:
  mov eax, 9 ;low end
  xor edx, edx ;high end is 0
  DIV QWORD [divisor] ;bp1 - RAX is 9

exit: mov rax, 60 ;bp2 - EAX is 4 and EDX is 1
      syscall


```

At the first breakpoint, register `EAX` has the low-order bits with the value `9` while register `EDX` has the high-order bits with the value `0`:

![](assets/posts/ASM/math/div3-1.png)


At the second breakpoint, register `EAX` has the quotient `4` while `EDX` with the remainder `1`:

![](assets/posts/ASM/math/div3-2.png)
