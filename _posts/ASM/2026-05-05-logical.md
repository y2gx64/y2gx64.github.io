## AND
The AND instruction is used for supporting logical expressions by performing bitwise AND operation. The bitwise AND operation returns 1, if the matching bits from both the operands are 1, otherwise it returns 0.

```
0001
0011
----
0001

```
### Example
```
segment .data  ;define variables and initialise them

segment .text
  global _start ;this tells the kernel to run from _start

_start:
  mov ah, 1; Example1
  mov al, 1
  AND ah, al
  NOP ; bp 1, ah should be 1

  mov ah, 3 
  mov al , 1
  AND ah,al
  NOP ; bp 2, ah should be 1

  mov ah, 2
  mov al, 1
  AND ah, al
  NOP ;bp 3, ah shold be 0

  mov rax, 60
  syscall
```
At breakpoint 1, i can see that `AH` has the value `1`. As AND `0x00000001` and `0x0000001`, the first bit will be `1` due to both registers having the value `1` in their first bits. This is stored in `AH` hence it having the value of `1`.

![](assets/posts/ASM/logical/AND_1.png)

At breakpoint 2, i can see that `AH` has the value `1`. As AND `0x00000011` and `0x0000001`, the first bit will be `1` due to both registers having the value `1` in their first bits while only `AH` having `1` in its second bit . This is stored in `AH` hence it having the value of `1`.

![](assets/posts/ASM/logical/AND_2.png)


At breakpoint 3, i can see that `AH` has the value `0`. As when AND `0x00000010` and `0x0000001`, none of the bits have matching `1` bits. This value is stored in `AH`, hence it having the value `0`.

![](assets/posts/ASM/logical/AND_3.png)

### TEST
The TEST instruction works same as the AND operation, but unlike AND instruction, it does not change the first operand. So, if we need to check whether a number in a register is even or odd, we can also do this using the TEST instruction without changing the original number.

```
TEST    AL, 01H
JZ      EVEN_NUMBER
```

## OR
The OR instruction is used for supporting logical expression by performing bitwise OR operation. The bitwise OR operator returns 1, if the matching bits from either or both operands are one. It returns 0, if both the bits are zero.
```
0x0010
0x0011
------
0x0011

```
### Example
```

segment .data  ;define variables and initialise them

segment .text
  global _start ;this tells the kernel to run from _start

_start:
  mov ah, 1; Example1
  mov al, 1
  OR ah, al
  NOP ; bp 1: ah should be 1

  XOR ah,ah
  XOR al,al 

  mov ah, 3
  mov al , 1
  OR ah,al
  NOP ; bp 2: ah should be 3, since 0x00000011
  
  XOR ah,ah
  XOR al,al

  mov ah, 4
  mov al, 3
  OR ah, al
  NOP ;bp 3: ah shold be 7, since 0x00000111
  
  XOR ah,ah
  XOR al,al

  mov rax, 60
  syscall

```
At breakpoint 1, i can see that `AH` has the value `1`. As OR `0x00000001` and `0x0000001`, the first bit will be `1` due to both registers having the value `1` in their first bits. This is stored in `AH` hence it having the value of `1`:

```
0x00000001
0x00000001
---------- 
0x00000001
```

![](assets/posts/ASM/logical/OR_1.png)


At breakpoint 2, i can see that `AH` has the value `3`. As OR `0x00000011` and `0x0000001`, as the second bit of `AH` has `1` while the first bits of both registers having the value `1`, this gives a result of `0x00000011` which is stored in `AH` as `3`.

```

0x00000011
0x00000001
----------
0x00000011
```

![](assets/posts/ASM/logical/OR_2.png)


At breakpoint 3, i can see that `AH` has the value `7`, As OR `0x00000100` and `0x00000011` with third bit of `AH` being `1` while the first and second bits of `AL` being `1`, this causes `AH` to have the value `7`.

```
0x00000100
0x00000011
---------- 
0x00000111
```

![](assets/posts/ASM/logical/OR_3.png)


##  XOR
The XOR instruction implements the bitwise XOR operation. The XOR operation sets the resultant bit to 1, if and only if the bits from the operands are different. If the bits from the operands are same (both 0 or both 1), the resultant bit is cleared to 0.

```
0x0010
0x0011
------ 
0x0001
```


### Example

```
segment .data  ;define variables and initialise them

segment .text
  global _start ;this tells the kernel to run from _start

_start:
  mov ah, 1; Example1
  mov al, 1
  XOR ah, al
  NOP ; bp 1: ah should be 0

  XOR ah,ah
  XOR al,al 

  mov ah, 3 
  mov al , 1
  XOR ah,al
  NOP ; bp 2: ah should be 2, since 0x00000010
  
  XOR ah,ah
  XOR al,al

  mov ah, 3
  mov al, 5
  XOR ah, al
  NOP ;bp 3: ah shold be 6, since 0x00000110
  
  XOR ah,ah
  XOR al,al

  mov rax, 60
  syscall

```

At breakpoint 1, i can see that `AH` has the value `0`. As XOR `0x00000001` and `0x0000001`, the first bit will be `0` due to both registers having the value `1` in their first bits which does not make it exclusive. Hence the value of `AH` is `0`.


```
0x00000001
0x00000001
-----------
0x00000001
```

![](assets/posts/ASM/logical/XOR_1.png)

At breakpoint 2, i can see that `AH` has the value `2`. As XOR `0x00000011` and `0x0000001`, the first bit will be `1` due to the value of the first bits of both registers being exclusive. Hence, the value of `AH` being `2`.

```
0x00000011
0x00000001
-----------
0x00000010
```
![](assets/posts/ASM/logical/XOR_2.png)


At breakpoint 3, i can see that `AH` has the value `6`. As XOR `0x00000011` and `0x0000101`, the first two bits will be `1` due to the value of the first three bits of both registers being exclusive. Hence, the value of `AH` being `6`.
```
0x00000011
0x00000101
-----------
0x00000110
```

![](assets/posts/ASM/logical/XOR_3.png)

## NOT
The NOT instruction implements the bitwise NOT operation. NOT operation reverses the bits in an operand. The operand could be either in a register or in the memory.


```
0x00000001
-----------
0x10000000
```
